# PRD V2 · Quota Butler - AI Agent Scheduler

## 一句话定义

Quota Butler V2 不再只是额度提醒器，而是 AI Agent Scheduler：

> 当用户还想继续工作时，下一个 Agent 已经准备好。

V2 的核心目标是最大化连续 AI 工作能力，而不是单纯最大化额度利用率。

---

## 当前已验证事实

2026-06-17 群内实测：

- 新飞书群：`额度管家提醒群`
- bot 可见群消息，能读取用户文字消息。
- 文字命令路由已验证：
  - `菜单` -> 返回测试菜单卡
  - `帮我安排明天` -> 返回平衡模式计划卡
  - `今天冲刺` -> 返回不断粮模式计划卡
  - `额度` -> 返回状态卡
  - 无关消息如 `哈哈` -> 静默，不回复
- 状态卡实测读到 Codex 当前额度；Claude Code token 过期时能展示失败原因。
- 卡片按钮已能展示，但点击提示“目标回调服务当前未在线”。

结论：

- 文字 router 可作为临时测试/兜底。
- 正式交互必须接通 `lark-channel-bridge` 或等价 callback bridge。

## 产品原则

### 1. Bridge 是正式交互闭环

正式路径必须支持卡片按钮点击：

```text
用户点击飞书卡片按钮
  -> 飞书 callback
  -> lark-channel-bridge
  -> quota_butler.handler
  -> 执行动作并回卡片
```

文字命令不是主交互，只保留为：

- bridge 挂掉时的兜底
- 用户主动查询入口
- 开发/联调工具

### 2. 主动推送是 V2 核心

系统不能只等用户问。需要在关键时间主动推送：

- 当前窗口恢复
- 某个 Agent 可用
- 用户未采用计划
- 当前不在安静时段
- 同一窗口未重复提醒

示例：

```text
Codex 已恢复，可以 one up 了

当前可用：Codex
Claude Code：token 过期
建议：现在启动 Codex，保持连续工作

[现在启动] [稍后提醒] [今天不提醒]
```

### 3. 卡片应是工作面板，不是日志

状态卡要视觉化：

- 明确显示 Agent 状态
- 用进度条表达额度百分比
- 区分可用、临界、过期、读取失败
- 错误信息要可行动，例如提示刷新 Claude 登录

计划卡要表达“人机协作节奏”：

- 机器节点：预热、恢复、可接力
- 人的创作窗口：自由发挥、深度工作、验证、整理
- 让用户看懂“明天怎么工作”，而不是只看到“几点谁启动”

---

## 核心指标

### CAS · Continuous Availability Score

工作时间内至少有一个 Agent 可用的时间比例。

```text
CAS = 至少一个 Agent 可用的时间 / 用户计划工作时间
```

V2 计划卡和主动推送都应围绕 CAS 表达。

## V2 分阶段范围

### V2-a · 群内交互可用

- 文字 router 可稳定响应
- 菜单卡、计划卡、状态卡可读
- 无关消息静默

当前已基本跑通。

### V2-b · Bridge 按钮闭环

- 初始化并启动 callback bridge
- 菜单按钮点击可触发 handler
- 按钮和文字命令结果一致

这是正式体验的必要条件。

### V2-c · 卡片视觉升级

- 状态卡加入进度条
- 计划卡改成协作时间线
- 错误状态更可行动

### V2-d · 采用计划

- 点击“采用计划”后写入 active plan
- 生成本地定时 warmup 任务
- 支持查看/取消计划

### V2-e · 主动推送

- 基于真实 usage 和 state 判断恢复窗口
- 无 active plan 时主动推送 one-up 卡
- 支持稍后提醒、今天不提醒、去重

## 当前交付状态（2026-06-18）

V2-b 到 V2-e 的仓库代码与自动化测试已经实现。私人 bridge fork 已由 launchd
常驻，按钮 callback 可进入 `quota_butler.handler` 并回飞书。当前产品工作集中在
V2-f 逐步规划交互，不回退或替换现有 bridge 方案。

## V2-f · 逐步规划交互

“帮我安排明天”不再立即返回技术计划，而是依次收集：

1. 任务类型：编码开发 / 内容创作 / 调研分析 / 混合任务。
2. 工作强度：轻量 / 正常 / 高强度。
3. 工作时间：飞书原生时间选择器，默认北京时间 09:00–17:00。
4. 偏好摘要：生成计划，或单独修改任务、强度、时间。

系统自动检测 Claude Code 与 Codex。任意一个可用即可生成计划；两者可用时自动组合。
最终计划卡只显示实际使用的 Agent，不显示其他 Agent 的失败原因。

计划卡不再展示 `CAS` 缩写，改为直接展示：

- 计划覆盖率
- 预计空档
- 预计接力次数
- 各节点的工作目的

卡片提供“采用计划 / 调整设置 / 仅提醒”。没有 `plan_version: 2` 的旧计划卡不可采用。
真实预热始终只允许 Codex；Claude Code 可以参与规划，但不会被自动预热。
