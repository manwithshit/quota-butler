# Quota Butler V3 产品需求文档

> 版本：V3.0  
> 日期：2026-06-19  
> 产品定位：运行在用户 Mac 上、通过飞书交互的 Claude Code / Codex 额度管家。

## 1. 产品概述

Quota Butler V3 不再要求用户填写日常场景、任务类型或工作强度，而是围绕三个逐级增强的核心能力展开：

1. **看得见**：随时查询 Claude Code 与 Codex 的额度状态。
2. **提醒得刚好**：额度窗口恢复时，在合适的时间询问用户是否需要预热。
3. **提前安排**：用户只需告诉系统明天的重度使用时间，系统依据本机 Agent 状态自动生成并执行预热计划。

一句话价值：

> 你告诉 Quota Butler 什么时候要重度使用 AI，它负责看额度、抓恢复窗口，并在你授权后提前准备好 Claude Code 或 Codex。

V3 的设计原则是“少问、自动判断、解释清楚”。复杂的额度窗口计算留在系统内部，用户只做必要决策。

## 2. 目标用户与场景故事

### 2.1 初级：我只想知道额度还剩多少

小林平时只用 Claude Code。开发到一半时，他不知道当前 5 小时额度还剩多少，也不想打开网页或运行命令。

他在飞书输入：

> 额度

Quota Butler 返回一张状态卡：

- 🟢 Claude Code · 5 小时窗口
- `████░░░░░░ 42%`
- 预计恢复时间：14:30
- 7 天额度：正常
- Codex：⚪ 未检测到安装

小林不需要理解 Token、窗口算法或内部指标，只需要一眼知道还能不能继续工作。

### 2.2 进阶：额度恢复了，但我不想被无意义地打扰

下午 14:30，Claude Code 的额度窗口恢复。Quota Butler 判断当前不在 23:00–08:00 的安静时段，于是发送一张简短卡片：

> ⚡ Claude Code 已恢复。接下来准备重点使用吗？

按钮：

- 立即预热
- 30 分钟后提醒
- 暂时不用

小林点击“立即预热”，即明确授权 Quota Butler 在本机向 Claude Code 发送一次真实预热请求。系统不再弹出第二次确认，只在执行后回复：

> ✅ Claude Code 已预热，新的额度窗口已开始。

如果恢复发生在夜间，Quota Butler 只记录状态，不发送消息，也不在早上补发已经过时的提醒。

### 2.3 高级：明天上午要连续工作，希望提前准备两个额度窗口

晚上 22:00，Quota Butler 发送当天唯一一张睡前询问卡：

> 🌙 明天有重度使用 AI 的计划吗？

按钮：

- 明天要重度使用
- 明天不用

小林选择“明天要重度使用”，然后只需选择：

- 一个开始时间；或
- 一个明确时间区间。

例如，他选择明天 09:00 开始。Quota Butler 优先围绕同一个 Agent 安排两个额度窗口：

- 提前启动第一个窗口；
- 让该窗口在重度工作期间恢复；
- 恢复后再启动同一 Agent 的第二个窗口。

系统最终展示一张短计划卡：

> 明天从 09:00 开始，优先使用 Claude Code。  
> 将在工作开始前准备第一个窗口，并在工作期间接上第二个窗口。

用户点击“采用计划”后，即授权计划卡中列出的全部 Claude Code / Codex 预热任务。系统分别创建本机定时任务，到点执行，不再逐个询问。

### 2.4 高级扩展：双 Agent 延长连续工作时间

阿文同时安装并登录了 Claude Code 与 Codex，需要从 09:00 连续工作到 18:00。

Quota Butler 不会一开始就把两个 Agent 混在一起，而是：

1. 优先让同一个 Agent 在前 5 小时内跨越两个额度窗口；
2. 当工作时间超过单 Agent 的稳定覆盖能力，或实际恢复窗口无法满足需求时；
3. 再把第二个 Agent 作为接力，延长整体连续可用时间。

计划卡说明：

> 当前检测到 Claude Code + Codex。  
> 前半段优先保持 Claude Code 连续工作，后半段由 Codex 接力，减少频繁切换。

双 Agent 是连续工作时间不足时的增强手段，不是默认混用策略。

## 3. 产品范围

### 3.1 V3 包含

- 飞书文字与菜单按钮查询额度。
- Claude Code / Codex 安装、登录与额度状态检测。
- 5 小时与 7 天额度可视化。
- Codex Token 过期后自动刷新一次并重试。
- Claude 登录失效时明确提示用户重新登录。
- 额度窗口恢复提醒。
- 23:00–08:00 安静时段。
- 每天 22:00 睡前计划询问。
- 恢复提醒与睡前询问合并。
- 重度使用开始时间或时间区间选择。
- 自动选择单 Agent 或双 Agent。
- 计划预览、采用、查看与取消。
- 用户明确授权后的 Claude Code / Codex 定时预热。

### 3.2 V3 删除

- 首次使用时填写日常场景。
- 编码开发、内容创作、调研分析、混合任务选择。
- 轻量、正常、高强度选择。
- 今天冲刺、不断粮模式、节省模式。
- CAS 等内部计算指标面向用户展示。
- V2 场景偏好与调度模式配置。

相关代码路径、状态字段、卡片与测试全部删除，不作为隐藏高级功能保留。

## 4. 功能需求

### 4.1 基础额度查询

正式入口：

- 文字：`额度`、`查看额度`、`quota`
- 菜单按钮：`当前额度`

两种入口返回同一张状态卡。

每个 Agent 展示：

- Agent 名称。
- 安装/连接状态。
- 5 小时窗口已用百分比及进度条。
- 预计恢复时间。
- 7 天窗口已用百分比（接口提供时）。
- 一句人话状态。

状态语义：

- 🟢 余量充足：已用低于 30%。
- 🟡 正常使用：已用 30%–69%。
- 🟠 注意消耗：已用 70%–89%。
- 🔴 接近耗尽：已用 90% 以上。
- 🟡 需要重新登录：已安装，但登录凭据失效。
- 🟠 暂时无法读取：已安装且可能已登录，但网络或额度接口失败。
- ⚪ 未检测到安装：本机没有发现对应 CLI。

一次网络错误不得被描述为“用户没有安装”。

### 4.2 Agent 检测

检测分为三层：

1. **安装检测**
   - 查找 `claude` / `codex` 可执行文件。
   - 支持 launchd PATH 与常见安装路径。
2. **登录检测**
   - Claude Code：使用 `claude auth status`，并读取现有凭据状态。
   - Codex：检查 Codex CLI 及认证文件是否可用。
3. **额度检测**
   - 调用对应额度接口。
   - 单独分类认证失败、网络失败、服务端错误与响应格式异常。

Agent 只有在额度成功读取后才进入自动调度候选。

若安装或登录状态正常，但额度接口临时失败：

- 状态卡显示“暂时无法读取”；
- 不把该 Agent 判定为未安装；
- 不自动采用包含该 Agent 的新计划。

### 4.3 Token 与登录恢复

#### Codex

- 额度请求返回 401 时，调用 Codex CLI 自动刷新一次。
- 刷新后重新读取认证信息并重试额度接口。
- 每次查询最多自动刷新一次，防止循环调用。

#### Claude Code

- 不使用 `claude -p` 刷新过期登录。
- 登录失效时显示“重新连接 Claude Code”及 `claude auth login` 指引。
- 登录恢复后，下一次轮询自动重新纳入检测。

### 4.4 额度恢复提醒

触发条件：

- 上一次记录的窗口已经到达恢复时间；
- 当前窗口利用率接近 0，说明新窗口尚未明显使用；
- 当前没有覆盖同一时间的生效计划；
- 当前不在静音或安静时段；
- 同一 Agent、同一窗口未提醒过。

普通提醒卡：

> ⚡ Codex 已恢复。接下来准备重点使用吗？

按钮：

- 立即预热
- 30 分钟后提醒
- 暂时不用

“立即预热”是最终授权。点击后直接执行对应 Agent 的真实请求，不再二次确认。

### 4.5 安静时段与睡前询问

- 安静时段固定为每天 23:00–次日 08:00。
- 安静时段内只更新状态，不发送额度恢复提醒。
- 08:00 后不补发已经过时的夜间恢复提醒。
- 每天 22:00 发送一次睡前计划询问。
- 每天最多一张睡前卡。

如果 22:00 附近同时满足额度恢复提醒：

- 合并成一张卡；
- 先展示当前 Agent 余量；
- 再询问“明天有重度使用计划吗？”；
- 不再发送单独恢复卡。

### 4.6 明日重度使用时间

用户点击“明天要重度使用”后，进入一个极简时间卡：

- `从某时开始`
- `指定时间区间`

选择“从某时开始”时：

- 用户只选择开始时间；
- 系统内部以“至少覆盖 5 小时连续工作，并尽量让同一 Agent 在此期间使用两个额度窗口”为最低优化目标；
- 只检测到一个可用 Agent 时，默认计划区间为 5 小时；
- 检测到两个可用 Agent 时，默认计划区间为 8 小时；
- 5 小时是最低保障目标，不是双 Agent 计划的最长时间；
- 计划卡会展示系统推导出的结束时间，用户可通过“调整时间”改为明确区间。

选择“指定时间区间”时：

- 用户选择开始与结束时间；
- 系统以完整覆盖该区间为目标；
- 时间区间最长 16 小时；
- 结束时间必须晚于开始时间。

### 4.7 调度规则

调度按以下优先级执行：

1. **真实窗口优先**
   - 使用当前额度接口返回的 `resets_at`，不盲猜整点。
2. **同一 Agent 优先**
   - 在连续工作的前 5 小时内，优先让同一 Agent 跨越两个额度窗口；
   - 减少 Claude Code / Codex 之间的上下文迁移。
3. **单 Agent 足够则不混用**
   - 只检测到一个 Agent 时，仅使用该 Agent；
   - 检测到两个 Agent，但单 Agent 已能满足时，也优先单 Agent。
4. **双 Agent 用于延长或补洞**
   - 用户指定超过 5 小时的区间；
   - 单 Agent 的真实恢复时间无法覆盖目标；
   - 或单 Agent 暂时不可用；
   - 此时才使用第二个 Agent 错峰接力。
5. **可执行性优先**
   - 计划生成与采用时都再次检查 Agent；
   - 不可用 Agent 不进入新计划。

单开始时间的典型目标：

- 若当前已有可用窗口且恢复点落在目标区间内，直接沿用该窗口；
- 若尚未形成有效窗口，优先在开始时间前约 2.5 小时预热，使恢复点落在前 5 小时中段；
- 在真实恢复点到达后预热同一 Agent 的第二个窗口；
- 若双 Agent 可用，可继续向后延长覆盖。

如果当前窗口状态无法让同一 Agent 在前 5 小时内跨越两个窗口：

- 系统不得伪造“已保障两个窗口”的结论；
- 应优先调整可执行的预热时间；
- 仍无法满足时，才引入第二个 Agent 补足空档；
- 计划卡必须解释使用双 Agent 的原因。

### 4.8 计划预览与授权

计划卡只展示用户需要理解的信息：

- 明天的工作时间。
- 当前检测到并采用的 Agent。
- 每个预热节点的时间与目的。
- 为什么使用单 Agent 或双 Agent。
- Claude Code 预热会产生一次真实 Claude Code 请求。

按钮：

- 采用计划
- 调整时间
- 仅提醒

点击“采用计划”即授权卡片中列出的全部 Claude Code / Codex 预热任务。

授权后：

- 不再针对每个任务二次确认；
- 每个 Agent、每个时间节点创建独立 launchd 任务；
- 单个任务失败不阻塞其他任务；
- 执行后发送简短回执；
- 用户可随时查看或取消整个计划。

### 4.9 预热执行

即时预热：

- Claude Code：执行一次 `claude -p <短预热消息>`。
- Codex：执行一次 `codex exec <短预热消息>`。

定时预热：

- 一个 Agent 的一个时间节点对应一个独立 launchd 任务。
- 任务执行前验证当前 active plan 与任务身份。
- 已执行任务不可重复执行。
- Agent 登录失效时跳过该任务并发送失败回执。
- 不使用预热命令尝试修复 Claude 登录。

### 4.10 菜单

用户菜单仅保留：

- 当前额度
- 明日计划
- 查看当前计划

有生效计划时，“查看当前计划”显示：

- 工作时间。
- 参与 Agent。
- 待执行/已执行节点。
- 取消计划按钮。

## 5. 数据与状态

V3 状态至少记录：

- 每个 Agent 最近一次额度快照。
- 每个 Agent 安装、登录与额度读取状态。
- 每个 Agent 最近提醒过的窗口。
- 延后提醒截止时间。
- 当天睡前卡是否已发送。
- 当前 proposed / active plan。
- 每个定时任务的执行状态。

不再保存：

- 日常使用场景。
- 任务类型。
- 工作强度。
- V2 scheduler mode。

状态写入必须支持并发保护，避免主轮询、卡片点击与定时任务同时写入造成覆盖。

## 6. V2 迁移

V3 首次运行时：

1. 识别 `plan_version < 3` 的生效计划。
2. 卸载其全部 launchd 任务。
3. 删除对应 plist 与任务日志索引。
4. 清空 active plan。
5. 删除旧 `schedule_profiles`。
6. 旧卡片点击后统一提示：“该卡片已失效，请重新打开菜单。”

迁移不得执行任何旧预热任务。

## 7. 非目标

V3 不做：

- 自动安装 Claude Code 或 Codex。
- 后台替用户完成 Claude OAuth 登录。
- 根据任务内容选择模型。
- 跨设备云同步。
- 团队共享额度池。
- 保证服务商额度策略永远不变。
- 为了追求覆盖率而频繁切换 Agent。

## 8. 成功指标

### 基础功能

- 用户输入“额度”后 10 秒内收到状态卡。
- 安装、登录、接口故障不会被混为同一种错误。
- 进度条与恢复时间可一眼理解。

### 智能提醒

- 同一窗口最多提醒一次。
- 23:00–08:00 无主动打扰。
- 每天 22:00 睡前卡最多一张。
- 提醒与睡前询问同时命中时只出现一张卡。

### 智能明日计划

- 用户最多经过“是否使用 → 时间选择 → 采用计划”三步。
- 单 Agent 优先在前 5 小时内安排两个额度窗口。
- 双 Agent 只在延长覆盖或补足空档时出现。
- 采用后所有任务可查看、执行、回执和取消。

## 9. 验收场景

1. 只安装并登录 Claude Code：查询卡只展示 Claude 可用；生成 Claude 单 Agent 计划。
2. 只安装并登录 Codex：Codex 401 自动刷新后查询成功；生成 Codex 单 Agent 计划。
3. 两个都可用，需求 4 小时：优先选择一个 Agent，不强行混用。
4. 两个都可用，需求 9 小时：优先同一 Agent 两窗口，再由第二 Agent 接力。
5. Claude 已安装但登录过期：显示需要重新登录，不误判未安装。
6. Codex 接口临时 5xx：重试后仍失败则显示暂时无法读取，不误判未安装。
7. 22:00 同时发生窗口恢复：只发送合并睡前卡。
8. 01:00 发生窗口恢复：不推送，08:00 后不补发旧提醒。
9. 点击“立即预热 Claude”：直接执行一次，不再确认，并回执结果。
10. 采用双 Agent 计划：分别创建 Claude 与 Codex 的独立 launchd 任务。
11. 取消计划：所有未执行任务全部卸载并删除。
12. V2 升级：旧计划、旧任务和旧偏好被清理，旧卡不能继续执行。

## 10. 视频叙事建议

### 第一幕：从“看不见”到“一眼看懂”

画面：在飞书输入“额度”。  
结果：Claude Code 与 Codex 的彩色进度条出现。

文案：

> 第一版，我只是想解决一个最简单的问题：我的 AI 额度到底还剩多少？

### 第二幕：从“自己盯着”到“恢复时提醒我”

画面：额度恢复卡出现，点击“立即预热”。

文案：

> 第二版，它不只会显示额度。窗口恢复时，它会在飞书问我：接下来要不要重点使用？我点一下，它就在家里的 Mac 上帮我预热。

### 第三幕：从“提醒工具”到“明日调度”

画面：22:00 睡前卡 → 选择明天 09:00 → 自动生成单 Agent/双 Agent 计划 → 采用。

文案：

> 第三版，我连预热时间都不用算。只要告诉它明天什么时候重度使用，它会根据电脑里实际可用的 Claude Code 和 Codex，优先给同一个 Agent 接上两个额度窗口；时间再长，才让另一个 Agent 接力。

收尾：

> 它不是帮我创造更多额度，而是把我已经付费的额度，安排在真正需要工作的时间。
